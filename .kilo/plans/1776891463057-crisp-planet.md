# Plan: Fix Planning Storage Format & Review Issues

## Problem Analysis

### Issue 1: Fragile Markdown Step Detection (FIX)

The step line parser uses `line.contains("Step ")` which can false-match on plan descriptions containing the word "Step" (e.g., "Refactor the Step builder class").

**Current state:** Plans are stored as `.md` files with metadata embedded as HTML comments. The parser has to reverse-engineer structured data from a human-readable format.

**Option A: Fix the markdown parser (minimal change)**
- Tighten the step line regex to match `**Step N:**` format exactly
- Pros: No format migration, existing plans stay valid, keeps human-readability
- Cons: Still fragile — any format change breaks parsing, markdown is fundamentally a presentation format not a data format

**Option B: Store plans as TOML with rendered markdown alongside (RECOMMENDED)**
- Save the structured data as `.toml` (machine-readable, lossless round-trip)
- Generate a `.md` view for human consumption (read-only, re-generated from TOML)
- Pros: Zero data loss, no parser fragility, TOML is human-readable too, serde handles serialization natively
- Cons: Requires migrating the storage layer, `file_path` field semantics change slightly

**Option C: Store plans as JSON**
- Pros: Simpler serde integration than TOML for nested structures
- Cons: Less human-readable/editable than TOML, doesn't align with project's TOML-first config convention

**Decision:** Option B. TOML is the right fit because:
1. The project already uses TOML for config (consistent convention)
2. `serde` + `toml` crate already in dependencies
3. Plans are a config-like data structure (key-value with nested arrays)
4. No need to write a custom parser at all — `toml::to_string_pretty()` and `toml::from_str()` handle everything
5. The rendered markdown is kept as a **view** method, not the storage format

### Issue 2: DB Mutex Held Across tokio::select! (EASY FIX → DEFER)

The `tokio::sync::Mutex` on `db_conn` is locked inside the cascade future. If the orchestrator branch wins, the future is dropped and the guard is released immediately (Rust RAII). The actual concern is that during normal cascade calls, the lock is held for the entire LLM inference duration, blocking other potential DB users.

**Assessment:** Easy to fix by restructuring the cascade call to release the lock before returning the result. However, the current codebase only has one DB user (the cascade call), so there's no actual contention. This is a **design improvement for future-proofing**, not a current bug.

**Action:** Defer — add to README as a known limitation.

### Issue 3: ListPlansTool Loading All Plans (DEFER)

Loading every plan from disk for `list_plans` could be slow with many plans. Pagination or caching would help.

**Assessment:** Plans are small files (typically < 1KB). Even 1000 plans would load in < 100ms. Not a real issue until proven otherwise.

**Action:** Defer — add to README as a future improvement.

## Implementation Plan

### Task 1: Refactor Plan Storage to TOML

**Files to modify:**
- `src/planning/mod.rs` — rewrite `save_plan`, `load_plan`, `list_plans`, remove `parse_markdown`
- `src/planning/types.rs` — ensure all types derive `Serialize`/`Deserialize` cleanly for TOML
- `src/tools/planning_tools.rs` — update `ListPlansTool` and `GetPlanTool` to use `render_markdown` for display
- `tests/planning_test.rs` — update tests for TOML format

**Changes:**

1. **Change file extension** from `.md` to `.toml` in `create_plan()`:
   ```rust
   let filename = format!("{}_{}.toml", plan.created_at.format("%Y%m%d_%H%M%S"), &plan.id[..8]);
   ```

2. **Rewrite `save_plan()`** to serialize via TOML:
   ```rust
   pub fn save_plan(&self, plan: &Plan) -> Result<()> {
       // Serialize a TOML-friendly struct (without file_path)
       let data = PlanData::from_plan(plan);
       let content = toml::to_string_pretty(&data)?;
       std::fs::write(&plan.file_path, content)?;
       Ok(())
   }
   ```

3. **Rewrite `load_plan()`** to deserialize via TOML:
   ```rust
   pub fn load_plan(&self, file_path: &Path) -> Result<Plan> {
       let content = std::fs::read_to_string(file_path)?;
       let data: PlanData = toml::from_str(&content)?;
       Ok(data.to_plan(file_path))
   }
   ```

4. **Create `PlanData` struct** (TOML-friendly, without `file_path`):
   ```rust
   #[derive(Serialize, Deserialize)]
   struct PlanData {
       id: String,
       task_id: String,
       title: String,
       status: PlanStatus,
       created_at: String,  // RFC3339
       updated_at: String,  // RFC3339
       steps: Vec<PlanStep>,
   }
   ```
   This avoids `PathBuf` in TOML (which doesn't serialize well) and handles datetime as strings.

5. **Keep `render_markdown()` as a read-only view method** — unchanged, used for human display.

6. **Remove `parse_markdown()`** — no longer needed.

7. **Update `list_plans()`** to filter by `.toml` extension instead of `.md`.

8. **Update `Plan::file_path`** — only set at runtime by `PlanManager`, never serialized.

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Existing `.md` plan files become unreadable | Plans are auto-created per task, not user-managed. No migration needed. |
| TOML serialization of `chrono::DateTime` | Use RFC3339 strings (`to_rfc3339()`/`parse_from_rfc3339()`) in `PlanData`. |
| `PathBuf` not TOML-friendly | Exclude from `PlanData`, set at runtime in `to_plan()`. |
